## 架構演進：LangGraph → 波特派（Botrun-pipe）

### 舊架構問題（LangGraph）

- 多代理、多工具（上網、讀檔、計算）協作
- 能處理複雜計算（老農保險等），保證「準」
- 但回應時間長達 **200 秒**（4分鐘），使用者體驗極差
- 單線程：一步做完才能做下一步
- 流程控制依賴 LLM 思考決定，如同「黑盒子」
- LLM 工具選擇不穩定，可能因提問差異選錯工具

### 三路平行技術實驗

| 方案 | 說明 | 速度 | 結果 |
|------|------|------|------|
| 綁定搜尋法（Grounding Search） | Gemini 2.5 Pro + 濃縮提示工程，一次呼叫完成搜尋與回答 | 45-50秒 | **最終採用** |
| 抓答分離法（Pre-fetching） | 拆分為「非同步抓取」與「即時問答」兩階段 | 22秒 | 驗證預處理潛力 |
| 綁定外源法（Vertex AI Search） | 鎖定特定官網/PDF 作為知識來源 | 預期<20秒 | 未來潛力 |

### 新架構特性（波特派 Botrun-pipe）

| 面向 | LangGraph（舊） | Botrun-pipe（新） |
|------|-----------------|-------------------|
| 核心架構 | 單線程，循序處理 | 平行積木式（parallel） |
| 流程控制 | LLM 決定下一步 | botrun-pipe.sh 腳本固定 |
| 執行效率 | 200 秒（多次 LLM 往返） | 45-50 秒（LLM 只在關鍵步驟出場） |
| 穩定性 | 不穩定、難除錯（黑盒子） | 穩定、易除錯（每步清晰可見） |

**核心哲學**：大模型是思考者，CLI 是行動者

## 提示工程

### 濃縮法

將 550 行核心提示詞精煉至 350 行，四大技巧：

1. **關鍵詞與縮寫系統**：冗長專有名詞→精簡縮寫（如「老年農民福利津貼」→「老農津貼」）
2. **階層式結構**：XML 標籤 + Markdown + 大括號 + 縮排，組織成樹狀階層
3. **邏輯符號化**：「!」=否定、「->」=因果，樹狀圖/狀態圖描述決策路徑
4. **範例導引（Few-shot Prompting）**：正面/反面 shot 引導輸出格式

### 元指令（Meta-instruction）設計原則

- 核心精神：「教 AI 如何釣魚，不是給魚」
- 不將政策條文寫死，而是建立「處理框架」和「思考路徑」
- 確保提示詞的通泛性，適應未來政策變動

### SYSTEM_PROMPT 關鍵模組

| 模組 | 功能 |
|------|------|
| 動態數據強制驗證原則 | 禁止直接使用內部知識回答具體數字，強制觸發搜尋工具查證 |
| 津貼組合強制分析演算法 | 三步驟：獨立資格拆解→組合規則審查→輸出驗證 |
| 情境感知與規則遵循 | 偵測使用者提供的情境→遵循為最高準則→覆寫自身知識 |
| 零號原則 | 情境識別與政策澄清（模糊查詢處理） |
| 最高真實性原則 | 外部即時查證結果 > 內部知識 |
| 公式先行原則 | 計算前必須：明示公式→展示代入→給出結果 |
| 獨立資格原則 | 每個個體的津貼資格獨立計算 |
| 管轄權驗證 | 強制區分【全國性】vs【地方性】政策 |
| SAFER 問答防火牆 | 藍隊防禦機制，處理非津貼查詢與攻擊 |

### 六步驟分析流程

```
Step000 前置驗證與破題（驗證使用者假設）
  ↓
Step001 補助項目識別（性質分類、計算方式、完整性檢查）
  ↓
Step002 排斥條件檢查（擇優、排斥、重複給付禁止）
  ↓
Step003 多重身份優化（單獨 vs 組合申請最大化）
  ↓
Step003.5-3.6 天數/金額標準驗證
  ↓
Step004 精確計算與驗算
```
