# 114年10月 — 品質驗測與技術重構

## 一、背景與初始假設

初期採用「波特人評分法」的「真心全快神準」維度評估 AI 服務品質。

- **核心假設**：社工等專業使用者的首要需求是資訊品質（真實性、更新性、全面性、準確性）
- **專案定位**：將「速度（快）」視為「加分項」，優先級排在資訊品質之後
- 在處理複雜計算（如老農保險、租屋補助）時調整技術架構，讓津貼 Bot 達到 7 分以上（滿分 10 分）的可靠水準

## 二、使用者驗證與回饋

**發現一：「快」的再定義 — 從「加分項」到「生命線」**

- 台灣天然災害頻繁，花蓮災害經驗讓團隊看見不同使用情境
- 第一線社工在分秒必爭的援助現場，需要快速掌握救災人力、緊急補助辦法等即時資訊
- 原系統回應時間長達 4 分鐘，無法滿足現場迫切需求
- 使用者痛點流程：等待 → 分心 → 遺忘 → 放棄
- 結論：「回應速度」不再是加分項，而是決定工具能否發揮作用的關鍵指標

**發現二：「準」的滿足點 — 品質已跨越「信任門檻」**

- 社工普遍認為舊版系統在資訊品質（真、新、全、準）上已達 7 分以上
- 已達到「願意信賴並採納其資訊」的門檻

**發現三：「省」的幕後價值 — 技術優化是使用者體驗的驅動力**

- 核心策略是從根本上降低後端資源消耗，提升前端反應速度

## 三、提示工程與技術架構演進

**（一）運算效率：以「結構化指令」降低 Token 消耗**

導入「提示工程濃縮法」，將核心提示詞由 550 行精煉至 350 行，成功將系統回應時間大幅縮短至 45 秒。應用技巧：

- **關鍵詞與縮寫系統**：將冗長專有名詞轉化為精簡縮寫（如「老年農民福利津貼」→「老農津貼」）
- **階層式結構**：運用 XML 標籤、Markdown 格式、大括號與縮排，組織成樹狀階層
- **邏輯符號化**：以「!」代表否定、「->」代表因果，導入樹狀圖/狀態圖描述決策路徑
- **範例導引（Few-shot Prompting）**：提供正面/反面 shot 範例引導輸出格式

**（二）未來適應性：以「元指令」確保通用性**

- 核心精神：「避免寫死、保持通用性」
- 遵循「元指令（Meta-instruction）」設計原則：教 AI 學會如何釣魚，而非直接給魚
- 為 AI 建立「處理框架」或「思考路徑」，確保提示詞的通泛性

**（三）技術架構演進：從 LangGraph 到 Botrun-pipe**

三路平行技術實驗：
1. **【最終方案】綁定搜尋法（Grounding Search）**：Gemini 2.5 Pro + 濃縮至 250 行精簡化提示工程，一次呼叫完成搜尋與回答
2. **【速度探索】抓答分離法（Pre-fetching）**：問答環節可壓縮至 22 秒
3. **【未來潛力】綁定外源法（Vertex AI Search）**：預期可壓至 20 秒以內

最終選擇「綁定搜尋法」實踐為「波特派（Botrun-pipe）」架構：
- 回應時間：45-50 秒
- 核心優勢：平行積木式架構，將 200 秒縮短至 50 秒以內（近 4 倍效能提升）
- 流程透明：定義在 botrun-pipe.sh 腳本中，每步清晰可見
- 確定性控制：大模型是思考者，CLI 是行動者

## 四、系統迭代與測試結果

**第一次測試（2025/10/29）：62 題，通過 51 題，通過率 82.3%**

六維度評估標準「真新全快省準」：
1. **真（真實性）**：零幻覺，資訊可追溯可驗證
2. **新（即時性）**：最新法規版本
3. **全（全面性）**：完整覆蓋問題每一層面
4. **快（快捷性）**：端到端回應 50 秒以內
5. **省（省資源）**：降低 API 成本與 GPU 負載
6. **準（準確性）**：精確邏輯推理能力

**關鍵問題點分析**：
- **真-金額錯誤**：新北市第一胎生育獎勵金誤答 2 萬（正確 3 萬）；房東違規罰鍰區間錯誤
- **真-資格條件錯誤**：育兒津貼年齡限制誤答「未滿5歲」（正確「0至未滿6歲」）
- **真-計算基準過時**：職災傷病給付使用過時投保薪資 45,800 元（正確 72,800 元）
- **新-資訊延遲**：年度更新政策數據未同步
- **全-遺漏項目**：遺漏勞保/國保生育給付、職災住院照護補助等
- **準-災難性案例**：將大學生租金補貼查詢誤導至教育部校外住宿補貼
