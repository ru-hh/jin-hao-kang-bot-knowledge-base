# 115年2月 — 模型升級與多輪對話上線

## 一、壓力測試環境建立（2/3）

**問題**：正式服務設有 Rate Limit，無法施加有效壓力；若強行對正式環境施壓，又會影響真實使用者。

**做法**：實作「雙服務獨立部署」架構。用同一份程式碼、同一個 Docker 映像，透過環境變數 `DISABLE_RATE_LIMIT=true` 控制，部署出第二個獨立的 Cloud Run 服務（`p271-subsidy-stress`），完全不影響正式版（`p271-subsidy`）。

| 面向 | 正式版（p271-subsidy） | 壓測版（p271-subsidy-stress） |
|------|----------------------|-------------------------------|
| Rate Limit | 啟用 | 停用 |
| 部署指令 | `sh/deploy-gcp.sh` | `sh/deploy-gcp.sh --stress` |
| 用途 | 正式服務 | 壓力測試專用 |
| 共用程式碼 | 同一份 | 同一份 |

**新增工具**：

- **壓測腳本** `sh/stress-test-no-limit.sh`：可設定並發數（`-c`）與請求總數（`-n`），自動產生 CSV 測試報告
- **健康檢查端點強化**：新增 `mode` 與 `rate_limit` 欄位，讓維運人員一眼辨別連的是正式版還是壓測版

**設計原則**：一套程式碼、兩個環境、零互相干擾。

## 二、Gemini 3 升級 + 引證全面改造（2/9）

本月技術含量最高的一次更新，三件事同步推進。

### （一）AI 模型升級：Gemini 2.5 Pro → Gemini 3 Flash Preview

| 項目 | 升級前 | 升級後 |
|------|--------|--------|
| 模型 | Gemini 2.5 Pro | Gemini 3 Flash Preview |
| 端點 | 區域端點（asia-east1） | 全球端點（Global Endpoint） |
| URL 格式 | `https://asia-east1-aiplatform.googleapis.com/...` | `https://aiplatform.googleapis.com/...` |
| 影響範圍 | — | 8 個檔案同步修改 |

**架構陷阱**：Gemini 3 不支援區域端點（如 `asia-east1`），只能用全球端點。這是一個隱性的破壞性變更——URL 格式改變涉及 8 個檔案的設定，必須全部同步修改確保一致，任何一個遺漏都會導致 API 呼叫失敗。

### （二）引證 URL 解析機制重構

**原始問題**：AI 透過 Google Search 找到的引證連結，原本是 Google 內部重定向 URL（點了要先繞一圈 Google 伺服器才到目的地）。

| 做法 | 原本 | 改良後 |
|------|------|--------|
| 策略 | 跟隨重定向（Follow Redirect） | 讀取 302 回應的 Location Header |
| HTTP 請求次數 | 每個連結多次 | 每個連結一次 |
| 執行方式 | 逐一處理 | 全部平行進行 |
| 政府網站擋機器人 | 常被擋 | 不觸發（只讀 Header，不實際訪問） |

### （三）引證來源過濾 + 格式簡化

- **過濾**：只保留 `.gov.tw` 政府官方網站，新聞、部落格全部過濾掉
- **格式**：引證只顯示 URL，移除原本的標題文字——因為標題是 AI 自動生成的，存在幻覺風險，純 URL 雖然簡單但 100% 準確

**同日另新增**：`--skip-build` 部署參數，讓建置好的映像可以直接重新部署，不用整個重跑編譯，解決「build 完但 push 中斷」的痛點。

## 三、多輪對話記憶正式上線（2/25）

### 背景與決策轉變

1 月以前的架構決策是「單輪精準查詢」，理由是資安考量（阻斷攻擊鏈）。多輪對話的核心模組 12 月就寫好了，但當時只能透過 `/test` 前綴觸發，是開發者驗證用的暫時機制，串接方要自己在每則訊息前加這個前綴，根本不實用。

### 正式上線機制

| 項目 | 開發階段（/test） | 正式上線 |
|------|-------------------|----------|
| 觸發方式 | 訊息前綴 `/test` | 隱式自動觸發 |
| 識別鍵 | 手動指定 | LINE userId 自動作為 Session Key |
| 串接方額外處理 | 需加前綴 | 不需要任何處理 |
| 開發用程式碼 | 約 80 行偵測 + Debug 區塊 | 全數移除 |

### 記憶規格

| 參數 | 設定 |
|------|------|
| 最大輪數 | 6 輪（問一次答一次 = 一輪） |
| 逾時清除 | 30 分鐘無互動自動清除 |
| 溢出策略 | 超過 6 輪自動淘汰最舊的一輪 |

### 上線驗證

執行三輪連續問答測試：

| 輪次 | 提問 | 驗證重點 |
|------|------|----------|
| 第 1 輪 | 「什麼是老農津貼」 | 基礎查詢 |
| 第 2 輪 | 「金額多少」 | 是否理解「金額」指的是老農津貼 |
| 第 3 輪 | 「去哪申請」 | 是否維持前兩輪的上下文 |

結果：AI 三輪都正確理解前後文，測試全數通過。

### Rate Limit 同步放寬

多輪對話一個問題可能要問 3～4 次，維持每分鐘 3 次的限制會讓功能根本用不了，因此同步放寬三倍：

| 限制項目 | 調整前 | 調整後 | 倍數 |
|----------|--------|--------|------|
| 使用者層 RPM | 3 次/分 | 9 次/分 | ×3 |
| 使用者層 RPD | 20 次/日 | 60 次/日 | ×3 |
| 系統層總量管制 | 不變 | 不變 | — |
